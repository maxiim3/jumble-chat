# Nostr Auth Guide

This document explains how Nostr login works in this project.

Goal: a junior developer should be able to read this file, open the relevant code, understand the current flow, and safely extend it.

## What the app does today

When a user lands on the app, they do **not** join the chat immediately.

They first see a blocking modal with two choices:

- `Rejoindre anonymement`
- `Se connecter avec Nostr`

Only after one of these flows succeeds do we open the app WebSocket and let the user enter the chat.

This gives us two session modes:

- `guest`: random username generated by the server + ostrich avatar
- `nostr`: pubkey + signed proof + profile metadata fetched from Nostr relays

## Very short mental model

Think of the login flow as 3 layers:

1. `App` shows the entry modal and reacts to state.
2. `SessionService` decides which connection flow to run.
3. `SocketClientService` opens the chat socket only when the session is ready.

For Nostr specifically, `NostrAuthService` prepares the identity data before the chat socket is opened.

## Files to read first

Read these files in this order:

1. `src/app/app.html`
2. `src/app/services/session.service.ts`
3. `src/app/services/nostr-auth.service.ts`
4. `src/app/services/socket-client.service.ts`
5. `shared/socket-message.model.ts`
6. `server/server.ts`

## Main responsibilities

### `src/app/app.html`

Owns the entry UI.

- If `session.isReady()` is false, show the blocking modal.
- If `session.isReady()` is true, show the chat.

This file is the best place to start if you want to understand the user-visible flow.

### `src/app/services/session.service.ts`

This is the orchestrator.

It owns the high-level state of the entry flow:

- `choosing`
- `connecting-guest`
- `connecting-nostr`
- `ready`

It also exposes:

- `profile`
- `error`
- `isBusy`
- `isReady`

This service decides:

- what happens when the user clicks anonymous login
- what happens when the user clicks Nostr login
- when the UI should fall back to the modal again

Important idea: `SessionService` does **not** talk directly to relays or parse WebSocket events. It delegates that work.

### `src/app/services/nostr-auth.service.ts`

This service is responsible for Nostr-specific logic only.

It does 4 things:

1. detect `window.nostr`
2. request the user's public key
3. ask the extension to sign a proof event
4. fetch the user's metadata (`kind: 0`) from Nostr relays

Then it returns a `SessionInitProfile` that the chat socket can use.

This separation is important: if you later change the Nostr implementation, most of the app should not need to know.

### `src/app/services/socket-client.service.ts`

This service handles the app's own chat WebSocket.

It does **not** decide whether the user is guest or Nostr. It just receives a ready-made `SessionInitProfile` and opens the socket.

Its flow is:

1. open socket
2. send `session_init`
3. wait for `session_ready`
4. hydrate local state (`currentAuthor`, `messages`, `countUsers`)

This is the service to modify if you change the app protocol.

### `shared/socket-message.model.ts`

This file is the contract between frontend and backend.

If a message shape changes, both Angular and Bun must still agree on it.

Important shared types:

- `SessionAuthor`
- `SessionInitProfile`
- `SessionInitMessage`
- `ChatMessage`
- `SocketMessageModel`
- `SocketEvent`

If you want to add a new event, start here.

### `server/server.ts`

This is the Bun server.

It now waits for `session_init` before treating a client as ready.

That means a socket can exist in 2 states:

- connected but not ready
- ready and allowed into the chat

This distinction is important. It is what lets the app show a blocking modal first and only join the chat once the chosen identity is prepared.

## Nostr concepts used here

### `window.nostr`

This comes from a browser extension implementing NIP-07.

In practice we use:

- `getPublicKey()`
- `signEvent()`

If `window.nostr` is missing, Nostr login cannot continue.

### Pubkey

A Nostr public key identifies a Nostr account.

In this project:

- the extension gives us the pubkey
- we store it in `SessionInitProfile.pubkey`
- we may include it in the public author snapshot when the user is connected with Nostr

### Signed proof event

We ask the extension to sign an event of kind `22242`.

That is inspired by NIP-42 authentication patterns.

Important: in the current implementation, this proof is created and sent, but the server does **not** verify it yet.

So today this is best understood as:

- product-level Nostr identity flow
- not yet full server-side cryptographic authentication

If you need stronger trust, server verification is the next important feature.

### `kind: 0`

In Nostr, `kind: 0` is profile metadata.

We query relays for it to get:

- display name
- fallback name
- profile picture

If no metadata is found, we still allow login.

Fallbacks:

- display name: `nostr-<pubkey-prefix>`
- avatar: ostrich avatar

### Relays

Relays are Nostr servers.

We currently query a short list in `NostrAuthService`:

- `wss://relay.damus.io`
- `wss://relay.primal.net`
- `wss://nos.lol`

The service uses `Promise.any()` because we only need one relay to answer with metadata.

## End-to-end flow

### Guest flow

1. User clicks `Rejoindre anonymement`
2. `SessionService.joinAnonymously()` runs
3. `SocketClientService.open({ authMode: 'guest' })` opens the app WebSocket
4. Client sends `session_init`
5. Server creates a guest author:
   - random display name
   - ostrich avatar
   - `authMode: 'guest'`
6. Server sends `session_ready`
7. Client marks the session as ready and displays the chat

### Nostr flow

1. User clicks `Se connecter avec Nostr`
2. `SessionService.connectWithNostr()` runs
3. `NostrAuthService.connect()`:
   - checks `window.nostr`
   - gets pubkey
   - signs a proof event
   - fetches `kind: 0` metadata
4. It returns a `SessionInitProfile`
5. `SocketClientService.open(profile)` opens the app WebSocket
6. Client sends `session_init`
7. Server builds the public author snapshot from the provided profile
8. Server sends `session_ready`
9. Client marks the session as ready and displays the chat

## Why there is both `SessionService` and `SocketClientService`

Because they solve different problems.

`SessionService` answers:

- Which login flow are we currently in?
- Are we loading?
- Should the modal be visible?
- What error should the user see?

`SocketClientService` answers:

- Is the app socket open?
- What messages have we received?
- Who is the current author according to the server?
- How many users are online?

This split makes the code easier to extend.

## Why the server creates a public `author` snapshot

Messages do not only contain a text body.

They also carry an `author` object:

- `sessionId`
- `displayName`
- `avatarUrl`
- `authMode`
- `pubkey?`

That is useful because the UI can render each message without having to look up some external user store.

It also means historical messages keep the author data they were created with.

## Current limitations

These are intentional and important to understand.

### 1. The server does not verify Nostr proof yet

The client sends `proof`, but the Bun server currently trusts the incoming Nostr profile payload.

This means a malicious client could fake some parts of the Nostr identity if they bypass the normal UI.

If you want real auth, the server must:

- issue or validate a challenge
- verify the signature
- confirm the pubkey matches the signed event

### 2. Metadata fetch is best effort

If relays do not answer, the login still succeeds with fallbacks.

This is good for UX, but it means profile quality depends on relay availability.

### 3. Relay list is hardcoded

That is acceptable for now, but you may later want:

- configuration
- relay health checks
- better timeout strategy

## How to safely add features

### Feature: verify Nostr proof on the server

Start in:

- `shared/socket-message.model.ts`
- `server/server.ts`

Likely steps:

1. decide the exact proof shape the server expects
2. parse it in `session_init`
3. verify signature + challenge
4. reject invalid sessions before setting `ws.data.ready = true`

### Feature: remember the last login choice

Start in:

- `src/app/services/session.service.ts`

Persist only the UI preference, not the full trusted identity.

Example safe approach:

- remember `guest` or `nostr`
- on reload, preselect the flow
- still redo the real login steps

### Feature: refresh Nostr profile later

Start in:

- `src/app/services/nostr-auth.service.ts`
- `src/app/services/socket-client.service.ts`
- `server/server.ts`

You would need a new protocol event, for example:

- `profile_update`

Do not silently mutate existing message authors unless you explicitly want retroactive UI changes.

### Feature: show a better Nostr error state

Start in:

- `src/app/services/session.service.ts`
- `src/app/services/nostr-auth.service.ts`
- `src/app/app.html`

Good improvement: return structured error categories instead of only plain strings.

Examples:

- extension missing
- user rejected signature
- relay timeout
- invalid proof

## Debugging checklist

If Nostr login fails, inspect these in order:

1. Is `window.nostr` available in the browser?
2. Does `getPublicKey()` resolve?
3. Does `signEvent()` resolve?
4. Do any relays answer with `kind: 0` metadata?
5. Does the app WebSocket connect?
6. Is `session_init` sent?
7. Does the server answer with `session_ready`?

If the modal stays open, the issue is usually in one of those steps.

## Rules of thumb for future changes

- If you change login UI state, start in `SessionService`.
- If you change Nostr-specific behavior, start in `NostrAuthService`.
- If you change socket handshake or events, start in `shared/socket-message.model.ts`.
- If you change who is considered a valid connected client, start in `server/server.ts`.
- If you add a new message shape, update both Angular and Bun in the same change.

## Suggested next improvements

Good next tasks for a junior dev after reading this file:

1. Add a proper disconnect button.
2. Add structured Nostr error types instead of free text strings.
3. Move relay URLs to config.
4. Verify Nostr proof on the server.
5. Add tests for more relay failure cases.

## Summary

The important idea is simple:

- the app does not enter the chat immediately
- it first prepares an identity
- then it opens the app WebSocket
- the server confirms the session with `session_ready`

Nostr is currently used for:

- getting a pubkey
- signing a proof event
- fetching profile metadata

But the current implementation is still one step away from full server-side Nostr authentication.
